What is Terraform?
Chapter 00 · Beginner Level
📖 Definition (English)
Terraform is an open-source Infrastructure as Code (IaC) tool created by HashiCorp. It lets you define, provision, and manage cloud infrastructure using a declarative configuration language (HCL). Instead of clicking buttons in a cloud console, you write code that describes the infrastructure you want — and Terraform builds it.
📖 வரையறை (தமிழ்): Terraform என்பது HashiCorp நிறுவனம் உருவாக்கிய ஒரு கருவி. இது "Infrastructure as Code" என்ற கொள்கையில் இயங்குகிறது. அதாவது, cloud-ல் servers, databases, networks போன்றவற்றை click செய்யாமல், code எழுதி create செய்யலாம். உங்கள் infrastructure-ஐ ஒரு blueprint போல் define செய்கிறீர்கள்.
Real-world Example
🏗️ Building a house analogy: Imagine you want to build 10 identical apartments. Instead of telling the contractor verbally each time, you write a blueprint once. Terraform is that blueprint for cloud infrastructure. You write it once, deploy it 100 times — AWS, Azure, GCP — all from the same code.

🏗️ வீடு கட்டும் உதாரணம்: 10 apartments கட்ட வேண்டும் என்றால், ஒரு blueprint வரைந்தால் போதும், பலமுறை கட்டலாம். Terraform அதே போல cloud-ல் infrastructure build செய்யும் ஒரு blueprint.
🔑 Key Terms
IaC – Infrastructure as Code HCL – HashiCorp Config Language Declarative – Describe the end state Idempotent – Same result every run
Declarative vs Imperative: You don't say "create a server, then attach a disk, then configure network". You say "I want this infrastructure to exist" — Terraform figures out the steps.

Declarative vs Imperative (தமிழ்): Bash script-ல் step-by-step சொல்வது imperative. Terraform-ல் "இந்த infrastructure வேண்டும்" என்று சொல்வது declarative.

Terraform Architecture
Chapter 02 · How It Works Internally
1
Terraform Core
The brain. Reads your .tf config files, compares desired state vs current state, builds an execution graph, and orchestrates the apply process.
Core — உங்கள் config files படிக்கும், plan செய்யும், apply செய்யும் — இது Terraform-இன் brain.
2
State File (terraform.tfstate)
Terraform's memory. Tracks every resource it manages. On next apply, it compares real infrastructure against this file to detect what needs to change.
State file = Terraform-இன் நினைவாற்றல். "என்ன create செய்தோம்" என்று இங்கே சேமிக்கப்படும்.
3
Provider Plugins
Adapters for each cloud/service. AWS provider knows how to talk to AWS API. Azure provider to Azure ARM. You can even have Kubernetes, GitHub, Datadog providers.
Provider = cloud API-ஓடு பேசும் translator. AWS provider AWS-ஓடு, Azure provider Azure-ஓடு communicate செய்யும்.
4
Configuration Files (.tf)
Your infrastructure blueprint. Written in HCL (HashiCorp Configuration Language). Defines resources, variables, outputs, and data sources.
Configuration files — உங்கள் infrastructure-ஐ describe செய்யும் blueprint files. HCL language-ல் எழுதப்படும்.
Core Concepts
Chapter 03 · Fundamentals You Must Know
①
init — Initialize
Downloads providers, sets up backend, prepares working directory. First command to run in any new Terraform project.
init — providers download செய்யும், project ready செய்யும். புதிய project-ல் முதல் command.
②
plan — Preview Changes
Dry run. Shows what Terraform will create, modify, or destroy. Never applies anything — just shows the plan. Always run before apply!
plan — "என்ன நடக்கும்" என்று preview காட்டும். Actually எதுவும் create/destroy ஆகாது. Apply-க்கு முன் எப்போதும் plan run செய்யுங்கள்.
③
apply — Execute
Actually creates/modifies/destroys infrastructure. Asks for confirmation (yes/no) unless you pass -auto-approve. Updates the state file after.
apply — actual-ஆக infrastructure create/change/delete செய்யும். Confirmation கேட்கும், "yes" type செய்தால் proceed ஆகும்.
④
destroy — Tear Down
Removes all resources Terraform manages. Useful for dev/test environments. Be careful in production!
destroy — Terraform manage செய்யும் அனைத்தையும் delete செய்யும். Dev/test environment-க்கு useful. Production-ல் கவனமாக!
⑤
Resource
The fundamental building block. An EC2 instance, S3 bucket, security group, RDS database — all are resources. Defined with resource "type" "name" {}
Resource — Terraform-ல் create செய்யும் ஒவ்வொரு item-உம் resource. EC2 instance, S3 bucket — அனைத்தும் resources.
⑥
Data Source
Read-only. Fetch info from existing infrastructure (not managed by Terraform). E.g., get the latest Amazon Linux AMI ID without hardcoding it.
Data source — Terraform manage செய்யாத existing infrastructure info-ஐ read செய்யும். E.g. latest AMI ID automatically fetch.
HCL Language
Chapter 04 · HashiCorp Configuration Language
HCL Basics — Building Blocks
HCL is human-readable, JSON-compatible, and uses blocks, arguments, and expressions. Think of it as JSON with superpowers — comments, functions, loops, conditions.
HCL = Human-readable config language. JSON போலவே உள்ளது, ஆனால் comments, loops, conditions support செய்யும். மனிதர்கள் எளிதாக படிக்கலாம்.
# Provider configuration terraform { required_providers { aws = { source = "hashicorp/aws" version = "~> 5.0" } } } provider "aws" { region = "ap-south-1" # Mumbai region — India }
# Variables — like function parameters variable "instance_type" { description = "EC2 instance type" type = string default = "t3.micro" } variable "environment" { type = string # "dev", "staging", "prod" }
# Resource — the core building block resource "aws_instance" "web_server" { ami = "ami-0c55b159cbfafe1f0" instance_type = var.instance_type # reference variable tags = { Name = "MyWebServer-${var.environment}" Environment = var.environment ManagedBy = "Terraform" } }
# Outputs — expose values after apply output "server_ip" { description = "Public IP of the web server" value = aws_instance.web_server.public_ip } # Data source — read existing resources data "aws_ami" "latest_amazon_linux" { most_recent = true owners = ["amazon"] filter { name = "name" values = ["amzn2-ami-hvm-*-x86_64-gp2"] } }
Real-world Example — E-commerce Startup
🛍️ Chennai-based e-commerce startup "ShopDesi" needs 3 environments: dev, staging, prod. With Terraform variables and workspaces, same .tf files create different-sized infra per environment. Dev = t3.micro (cheap), Prod = t3.xlarge (powerful). One codebase, three environments.

Chennai startup "ShopDesi" — dev, staging, prod — மூன்று environments-க்கும் same Terraform code. Variable-ல் instance type மாறும். Dev-ல் cheap server, prod-ல் powerful server. ஒரே code, மூன்று environments!
State Management
Chapter 05 · terraform.tfstate — The Heart of Terraform
What is State?
Terraform state is a JSON file that maps your config to real-world resources. It stores IDs, metadata, and attribute values of every resource Terraform manages. Without state, Terraform can't know what already exists.
State file — Terraform-இன் memory. "AWS-ல் i-0abc123 என்ற instance create செய்தோம், அது web_server resource" என்று இங்கே track செய்யும். இதன் மூலம் Terraform என்ன already உள்ளது, என்ன change செய்யணும் என்று தெரியும்.
❌ Local State (Bad for Teams)
Only on your laptop
Team can't share it
No locking — concurrent apply = disaster
Lost if disk fails
✅ Remote State (Team-ready)
Stored in S3 / GCS / Azure Blob
Team shares same state
State locking via DynamoDB
Versioned and backed up
# Remote backend — store state in S3 (Best Practice) terraform { backend "s3" { bucket = "my-terraform-state-bucket" key = "prod/terraform.tfstate" region = "ap-south-1" encrypt = true # DynamoDB for state locking dynamodb_table = "terraform-state-lock" } }
State Commands
terraform state list — List all resources in state
terraform state show aws_instance.web — Show resource details
terraform state mv — Rename/move resource in state
terraform state rm — Remove resource from state (not from cloud)
terraform import — Import existing resource into state
State commands: list → எல்லா resources பார்க்க | show → ஒரு resource details | mv → rename | rm → state-ல் இருந்து நீக்க (cloud-ல் இருக்கும்) | import → already உள்ள resource-ஐ Terraform-க்கு add செய்ய
⚠️ Never manually edit terraform.tfstate! Corrupt state = Terraform loses track of your entire infrastructure. Always use terraform state commands. | ⚠️ terraform.tfstate-ஐ manually edit செய்யாதீர்கள். State corrupt ஆனால் Terraform-க்கு infrastructure பற்றி எதுவும் தெரியாமல் போகும்.
Modules
Chapter 06 · Reusable Infrastructure Components
What is a Module?
A module is a container for multiple resources used together. Think of it like a function in programming — write once, call many times with different parameters. Modules enable DRY (Don't Repeat Yourself) infrastructure code.
Module = programming-ல் function போல். ஒரு module-ல் பல resources define செய்யலாம், பலமுறை call செய்யலாம். "VPC module" ஒரு முறை எழுதி, dev, staging, prod — எல்லா environment-லயும் use செய்யலாம்.
# Using a module — like calling a function module "vpc" { source = "terraform-aws-modules/vpc/aws" version = "~> 5.0" name = "my-vpc" cidr = "10.0.0.0/16" azs = ["ap-south-1a", "ap-south-1b"] private_subnets = ["10.0.1.0/24", "10.0.2.0/24"] public_subnets = ["10.0.101.0/24", "10.0.102.0/24"] enable_nat_gateway = true } # Reference module outputs resource "aws_instance" "app" { subnet_id = module.vpc.private_subnets[0] }
Providers
Chapter 07 · Connecting to Cloud Services
What is a Provider?
Providers are plugins that Terraform uses to interact with cloud platforms, SaaS tools, and APIs. Each provider adds a set of resource types and data sources. There are 3,000+ providers in the Terraform Registry.
Provider = Terraform-க்கும் cloud/service-க்கும் இடையிலான translator plugin. AWS provider, Azure provider, Kubernetes provider, GitHub provider — இவை எல்லாம் Terraform-க்கு different services-ஐ manage செய்ய help செய்யும்.
# Multi-provider setup — AWS + GitHub + Datadog terraform { required_providers { aws = { source = "hashicorp/aws" version = "~> 5.0" } github = { source = "integrations/github" version = "~> 6.0" } datadog = { source = "DataDog/datadog" version = "~> 3.0" } } } # AWS provider — multiple regions using aliases provider "aws" { alias = "mumbai" region = "ap-south-1" } provider "aws" { alias = "us_east" region = "us-east-1" } resource "aws_s3_bucket" "backup" { provider = aws.us_east # use US region bucket = "my-backup-bucket" }
Popular Providers
hashicorp/aws — Amazon Web Services hashicorp/azurerm — Microsoft Azure hashicorp/google — Google Cloud hashicorp/kubernetes — K8s clusters hashicorp/helm — Helm charts integrations/github — GitHub repos DataDog/datadog — Monitoring cloudflare/cloudflare — DNS/CDN
AWS, Azure, GCP போன்ற cloud providers மட்டுமல்ல — GitHub, Datadog, Cloudflare, MongoDB Atlas, Kubernetes — எல்லாவற்றையும் Terraform manage செய்யலாம்!
Workspaces
Chapter 08 · Multiple Environments, One Codebase
What are Workspaces?
Workspaces allow multiple state files from the same configuration. Each workspace has its own isolated state. Useful for managing dev/staging/prod without duplicating code. Default workspace is always "default".
Workspace = same code, different environments. dev workspace-ல் dev state file, prod workspace-ல் prod state file. ஒரே .tf files-ஐ பயன்படுத்தி multiple environments manage செய்யலாம்.
# Workspace commands $ terraform workspace list # show all workspaces $ terraform workspace new dev # create dev workspace $ terraform workspace new staging $ terraform workspace new prod $ terraform workspace select prod # switch to prod $ terraform workspace show # current workspace
# Use workspace name in resources locals { env = terraform.workspace # "dev", "staging", "prod" instance_sizes = { dev = "t3.micro" staging = "t3.medium" prod = "t3.xlarge" } } resource "aws_instance" "app" { instance_type = local.instance_sizes[local.env] tags = { Environment = local.env } }
Real-world: Ola Cabs / Swiggy Infra
🚕 Swiggy has 4 environments: dev (developers test features), QA (testing team verifies), staging (production mirror), prod (live users). All 4 use identical Terraform code but different workspaces. Prod uses 50 EC2 instances; dev uses 2. Same codebase, workspace handles the difference!

Swiggy — dev, QA, staging, prod — நான்கு environments. Same Terraform code, different workspaces. Prod-ல் 50 servers, dev-ல் 2 servers. Workspace மட்டும் மாற்றினால் போதும்!
Backend & Remote State
Chapter 09 · Team Collaboration at Scale
Backend Types
Backends determine where Terraform stores its state and where operations run. Local backend (default) keeps state on disk. Remote backends store in cloud storage and support collaboration features like state locking.
Backend = state எங்கே store ஆகும் என்று decide செய்யும். Local backend = உங்கள் laptop-ல். Remote backend = S3/Azure Blob/GCS-ல். Team work-க்கு remote backend mandatory.
# Terraform Cloud / HCP Terraform backend terraform { cloud { organization = "my-company" workspaces { name = "production" } } }
S3
AWS S3 + DynamoDB
Most common for AWS shops. S3 stores state file (encrypted, versioned). DynamoDB table provides state locking (only one person applies at a time). Free tier friendly.
AWS S3-ல் state file, DynamoDB-ல் locking. ஒரே நேரத்தில் 2 பேர் apply செய்தால் conflict வராது.
TFC
Terraform Cloud (HCP Terraform)
HashiCorp's managed solution. Handles state, locking, remote plan/apply, secret management, policy enforcement, and audit logs. Free tier for small teams.
HashiCorp-இன் managed service. State, locking, CI/CD, secrets — எல்லாமே ஒரே platform-ல். Small teams-க்கு free.

Best Practices
Chapter 11 · Write Production-Grade Terraform
①
Structure Your Code
Separate files for logic: main.tf (resources), variables.tf (var definitions), outputs.tf (outputs), versions.tf (provider versions), terraform.tfvars (variable values).
Code separate files-ல் வை: main.tf, variables.tf, outputs.tf, versions.tf. ஒரு file-ல் எல்லாம் போட்டால் maintain செய்வது கஷ்டம்.
②
Never Hardcode Secrets
Use environment variables, AWS Secrets Manager, Vault, or SSM Parameter Store. Never put passwords/keys in .tf files. Add *.tfvars to .gitignore.
Passwords, API keys — .tf files-ல் hardcode செய்யாதீர்கள். Environment variables, AWS Secrets Manager use செய்யுங்கள். *.tfvars-ஐ .gitignore-ல் வையுங்கள்.
③
Use Remote State Always in Teams
Local state in teams = disaster. Always use S3+DynamoDB or Terraform Cloud. Enable versioning on S3 bucket for state recovery.
Team-ல் local state use செய்யாதீர்கள். S3+DynamoDB அல்லது Terraform Cloud mandatory. S3 bucket versioning enable செய்யுங்கள்.
④
Tag Everything
Add tags to every resource: Environment, Team, Project, ManagedBy=Terraform, Owner. This helps with cost allocation and troubleshooting.
Every resource-க்கும் tags add செய்யுங்கள். Environment, Team, Cost-center tags மூலம் billing track செய்யலாம், troubleshoot செய்யலாம்.
⑤
Pin Provider Versions
Always pin provider versions (~> 5.0 not latest). Breaking changes in providers can destroy your infrastructure if you auto-upgrade.
Provider version pin செய்யுங்கள். "~> 5.0" என்று fix செய்யுங்கள். Latest use செய்தால் breaking changes வந்து infrastructure break ஆகலாம்.
⑥
Run terraform fmt & validate
Before committing, always run terraform fmt (format code) and terraform validate (syntax check). Add these to pre-commit hooks and CI pipelines.
Commit-க்கு முன் terraform fmt (formatting) மற்றும் terraform validate (syntax check) run செய்யுங்கள். Pre-commit hooks-ல் add செய்யுங்கள்.
Master Level Tips
Chapter 12 · Expert-Level Terraform
Advanced Concepts
At master level, you go beyond basic resource management to dynamic configurations, policy enforcement, and custom providers.
# Dynamic blocks — avoid repetition resource "aws_security_group" "web" { dynamic "ingress" { for_each = var.ingress_rules content { from_port = ingress.value.port to_port = ingress.value.port protocol = "tcp" cidr_blocks = ingress.value.cidrs } } } # for_each — create multiple resources resource "aws_s3_bucket" "environments" { for_each = toset(["dev", "staging", "prod"]) bucket = "myapp-${each.key}-bucket" } # count — simple repetition resource "aws_instance" "workers" { count = 5 # creates 5 EC2 instances instance_type = "t3.micro" tags = { Name = "worker-${count.index}" } }
# Conditional expressions resource "aws_db_instance" "main" { multi_az = terraform.workspace == "prod" ? true : false deletion_protection = terraform.workspace == "prod" ? true : false backup_retention_period = terraform.workspace == "prod" ? 7 : 1 } # Sentinel / OPA — Policy as Code # Enforce: all S3 buckets must be private # Enforce: prod instances must be >= t3.medium # Enforce: all resources must have required tags
⚡
Terragrunt
Wrapper around Terraform that helps keep configurations DRY across multiple environments. Handles dependencies between Terraform modules automatically.
Terragrunt — Terraform wrapper. Multiple environments-க்கு same config duplicate செய்யாமல், DRY principle follow செய்ய help செய்யும்.
🔒
Policy as Code (OPA / Sentinel)
Enforce organizational policies before resources are created. "No public S3 buckets", "Prod must use encrypted disks", "Tags are mandatory" — enforced automatically.
Policy as Code — "public S3 bucket create செய்யக் கூடாது", "prod-ல் encryption mandatory" — இவற்றை automatically enforce செய்யும். Compliance automatically.
🧪
Terraform Test (v1.6+)
Native testing framework for Terraform modules. Write .tftest.hcl files to unit test your modules — assert on outputs, check resource configurations.
Terraform native testing framework. Module-ஐ test செய்யலாம். "VPC-ல் correct CIDR block உள்ளதா?" என்று assert செய்யலாம்.
📊
Infracost — Cost Estimation
Shows cloud cost breakdown before terraform apply. Integrate in CI/CD to see "this PR will increase AWS bill by ₹15,000/month" in PR comments.
Infracost — apply-க்கு முன்னே "இந்த infrastructure-க்கு ₹15,000/month cost ஆகும்" என்று PR comment-ல் காட்டும். Budget control automatic!
