Chapter 00 · Beginner
What is Terraform?
Definition, purpose and the big picture
📖 Definition — English
Terraform is an open-source Infrastructure as Code (IaC) tool built by HashiCorp. It lets you define, provision, and manage cloud infrastructure using a simple declarative language called HCL (HashiCorp Configuration Language). Instead of clicking buttons in the AWS/Azure/GCP console, you write code describing the infrastructure you want — and Terraform builds it for you.
📖 வரையறை — தமிழ்: Terraform என்பது HashiCorp நிறுவனம் உருவாக்கிய ஒரு open-source கருவி. Cloud-ல் servers, databases, networks போன்றவற்றை manually click செய்யாமல், code எழுதி automatically create செய்யலாம். உங்கள் infrastructure-ஐ ஒரு blueprint போல் describe செய்கிறீர்கள் — Terraform அதை build செய்யும்.
🏗️ Real-world Analogy · நிஜ உலக உதாரணம்
Building Blueprint: Imagine building 10 identical apartments. Instead of telling the contractor verbally each time, you write a blueprint once and reuse it. Terraform is that blueprint for cloud infrastructure — write it once, deploy it 100 times across AWS, Azure, and GCP.

வீடு கட்டும் உதாரணம்: 10 apartments கட்ட வேண்டும் என்றால், ஒரு blueprint வரைந்தால் போதும். பலமுறை கட்டலாம். Terraform அதே போல் cloud-ல் infrastructure build செய்யும் blueprint. ஒரே code — AWS, Azure, GCP எல்லா இடத்திலும் use செய்யலாம்!
🔑 Key Terms
IaC — Infrastructure as Code HCL — HashiCorp Config Language Declarative — Describe the end state Idempotent — Same result every run
Declarative vs Imperative: In Bash, you say "create server, attach disk, configure network" step-by-step (imperative). In Terraform, you say "I want this infrastructure to exist" and it figures out the steps (declarative).

Declarative (தமிழ்): Bash-ல் step-by-step சொல்வது imperative. Terraform-ல் "இந்த infrastructure வேண்டும்" என்று சொல்வது declarative — steps Terraform figure out செய்யும்.

Chapter 02 · Beginner
Terraform Architecture
How it works internally — Core, State, Providers
1
Terraform Core
The brain. Reads .tf config files, compares desired state vs current state, builds a dependency graph, and orchestrates the apply process in the right order.
Core = Terraform-இன் brain. Config files படிக்கும், plan செய்யும், dependency graph build செய்யும், apply orchestrate செய்யும்.
2
State File (terraform.tfstate)
Terraform's memory. A JSON file that maps your code to real-world resources. Stores IDs, metadata, and attributes of every managed resource. Next apply compares this to detect changes.
State file = Terraform-இன் நினைவாற்றல். "AWS-ல் i-0abc123 என்ற instance உள்ளது, அது web_server resource" — இப்படி track செய்யும். இதனால் change detect possible.
3
Provider Plugins
Adapters that know how to talk to each cloud/service API. AWS provider → AWS API. Azure provider → Azure ARM API. 3000+ providers available including GitHub, Datadog, Cloudflare.
Provider = cloud API-ஓடு பேசும் translator. AWS, Azure, GCP மட்டுமல்ல — GitHub, Datadog, Kubernetes — எல்லாவற்றுக்கும் providers உள்ளன.
4
Configuration Files (.tf)
Your infrastructure blueprint written in HCL. Contains resources, variables, outputs, data sources, and module calls. Terraform reads these to know what infrastructure to create.
Configuration files — HCL language-ல் எழுதப்படும் infrastructure blueprint. Resources, variables, outputs — எல்லாமே இங்கே define செய்யப்படும்.

①
terraform init — Initialize
Downloads providers, sets up backend, prepares the working directory. Run this first in any new Terraform project. Safe to run multiple times.
init — providers download, backend setup, working directory prepare. புதிய project-ல் முதல் command. பலமுறை run செய்யலாம்.
②
terraform plan — Dry Run Preview
Shows what will be created (+), modified (~), or destroyed (-). Never applies anything. Always run before apply to review changes. Save with -out=tfplan for safe apply.
plan — என்ன நடக்கும் என்று preview. Actually எதுவும் create/destroy ஆகாது. Apply-க்கு முன் எப்போதும் plan run செய்யுங்கள்!
③
terraform apply — Execute Changes
Actually creates/modifies/destroys infrastructure based on the plan. Asks for confirmation. Use -auto-approve in CI/CD. Updates state file after completion.
apply — actual-ஆக infrastructure create/change/delete. "yes" type செய்தால் proceed. State file automatically update ஆகும்.
④
terraform destroy — Tear Down All
Removes all Terraform-managed resources. Useful for dev/test environments to save costs. Always double-check before running in production!
destroy — Terraform manage செய்யும் அனைத்தையும் delete. Dev/test-ல் cost save செய்ய useful. Production-ல் மிகவும் கவனமாக!
⑤
Resource — The Building Block
Every piece of infrastructure is a resource. An EC2 instance, S3 bucket, security group, RDS database, VPC — all are resources. Defined with resource "type" "name" {}
Resource = Terraform create செய்யும் ஒவ்வொரு item. EC2, S3, VPC, DB — அனைத்தும் resources. resource block-ல் define செய்யப்படும்.
⑥
Data Source — Read-Only Lookup
Fetch info from existing infrastructure not managed by Terraform. E.g., get the latest Amazon Linux AMI ID automatically without hardcoding it.
Data source — Terraform manage செய்யாத existing infrastructure info read செய்யும். Latest AMI ID, existing VPC details — automatically fetch.
Chapter 04 · Intermediate
HCL Language
HashiCorp Configuration Language — syntax, blocks, expressions
What is HCL?
HCL is human-readable, JSON-compatible, and supports comments, functions, loops, and conditions. Think of it as JSON with superpowers — designed for humans to write and machines to parse.
HCL = Human-readable config language. JSON போலவே, ஆனால் comments, loops, conditions support. மனிதர்கள் எளிதாக படிக்கும் வகையில் design செய்யப்பட்டது.
# 1. Provider configuration — tell Terraform which cloud terraform { required_providers { aws = { source = "hashicorp/aws" version = "~> 5.0" # pin to v5.x } } } provider "aws" { region = "ap-south-1" # Mumbai region }
# 2. Variables — like function parameters variable "instance_type" { description = "EC2 instance type" type = string default = "t3.micro" } variable "environment" { type = string description = "dev | staging | prod" validation { condition = contains(["dev","staging","prod"], var.environment) error_message = "Must be dev, staging, or prod." } }
# 3. Resource — the core building block resource "aws_instance" "web_server" { ami = data.aws_ami.latest.id # from data source instance_type = var.instance_type tags = { Name = "WebServer-${var.environment}" Environment = var.environment ManagedBy = "Terraform" } } # 4. Output — expose values after apply output "server_ip" { description = "Public IP of web server" value = aws_instance.web_server.public_ip } # 5. Data source — read existing AMI data "aws_ami" "latest" { most_recent = true owners = ["amazon"] filter { name = "name" values = ["amzn2-ami-hvm-*-x86_64-gp2"] } }
🛍️ Real-world: ShopDesi Startup — Chennai
Chennai e-commerce startup "ShopDesi" needs dev, staging, and prod environments. Same .tf files, different variable values. Dev = t3.micro (cheap), Staging = t3.medium, Prod = t3.xlarge. One codebase, three environments!

"ShopDesi" — dev, staging, prod — same Terraform code. Variable-ல் instance type மாறும். Dev-ல் cheap server, prod-ல் powerful server. ஒரே code, மூன்று environments!
Chapter 05 · Intermediate
State Management
terraform.tfstate — the heart of Terraform
What is Terraform State?
Terraform state is a JSON file that maps your configuration to real-world resources. It tracks IDs, metadata, and attributes of every managed resource. Without state, Terraform can't know what already exists in your cloud account.
State file = Terraform-இன் memory. "AWS-ல் i-0abc123 instance உள்ளது — அது web_server resource" என்று இங்கே track ஆகும். அடுத்த apply-ல் what changed என்று detect செய்ய இது help செய்யும்.
❌ Local State (Solo only)
Only on your laptop
Team can't share
No locking — 2 people apply = disaster
Lost if disk fails
✅ Remote State (Team-ready)
Stored in S3 / GCS / Blob
Whole team shares one state
DynamoDB locking — safe concurrent use
Versioned & backed up
# Remote backend — S3 + DynamoDB (AWS best practice) terraform { backend "s3" { bucket = "my-terraform-state-prod" key = "services/web/terraform.tfstate" region = "ap-south-1" encrypt = true dynamodb_table = "terraform-state-lock" # prevents concurrent apply } }
Essential State Commands
terraform state list— List all managed resources
terraform state show aws_instance.web— Full resource details
terraform state mv old_name new_name— Rename in state
terraform state rm aws_instance.web— Remove from state only
terraform import aws_instance.web i-0abc— Adopt existing resource
State commands: list → எல்லா resources | show → resource details | mv → rename | rm → state-ல் இருந்து நீக்கு (cloud-ல் இருக்கும்) | import → existing resource-ஐ Terraform-க்கு add செய்
⚠️ Never manually edit terraform.tfstate! A corrupt state file means Terraform loses track of all infrastructure. Always use terraform state commands.
⚠️ terraform.tfstate-ஐ manually edit செய்யாதீர்கள். State corrupt ஆனால் Terraform-க்கு infrastructure பற்றி எதுவும் தெரியாமல் போகும்!
Chapter 06 · Intermediate
Modules
Reusable infrastructure components — DRY principle
What is a Module?
A module is a container for multiple resources used together — like a function in programming. Write once, call many times with different parameters. Enables DRY (Don't Repeat Yourself) infrastructure code.
Module = programming-ல் function போல். ஒரு module-ல் பல resources define செய்யலாம், பலமுறை different parameters-உடன் call செய்யலாம். DRY principle — ஒரே code, multiple uses.
# Using a public module from Terraform Registry module "vpc" { source = "terraform-aws-modules/vpc/aws" version = "~> 5.0" name = "shopDesi-vpc-${var.environment}" cidr = "10.0.0.0/16" azs = ["ap-south-1a", "ap-south-1b", "ap-south-1c"] private_subnets = ["10.0.1.0/24", "10.0.2.0/24"] public_subnets = ["10.0.101.0/24", "10.0.102.0/24"] enable_nat_gateway = true } # Reference module outputs in other resources resource "aws_instance" "app" { subnet_id = module.vpc.private_subnets[0] # from module output }
Chapter 07 · Intermediate
Providers
Connecting Terraform to cloud services & APIs
What is a Provider?
Providers are plugins that know how to interact with cloud platforms, SaaS tools, and APIs. Each provider adds resource types and data sources. 3,000+ providers exist in the Terraform Registry — AWS, Azure, GCP, Kubernetes, GitHub, Datadog, Cloudflare, MongoDB, and more.
Provider = Terraform-க்கும் cloud/service-க்கும் இடையிலான translator plugin. AWS provider = AWS API-ஓடு பேசும். GitHub provider = GitHub-ஓடு பேசும். 3000+ providers available!
# Multi-provider setup — AWS + GitHub + Kubernetes terraform { required_providers { aws = { source = "hashicorp/aws", version = "~> 5.0" } github = { source = "integrations/github", version = "~> 6.0" } kubernetes = { source = "hashicorp/kubernetes", version = "~> 2.0" } } } # Provider aliases — manage multiple AWS regions provider "aws" { alias = "mumbai" region = "ap-south-1" } provider "aws" { alias = "us_east" region = "us-east-1" } resource "aws_s3_bucket" "dr_backup" { provider = aws.us_east # disaster recovery in US bucket = "shopDesi-disaster-recovery" }
Popular Providers
hashicorp/aws hashicorp/azurerm hashicorp/google hashicorp/kubernetes hashicorp/helm integrations/github DataDog/datadog cloudflare/cloudflare mongodb/mongodbatlas
AWS, Azure, GCP மட்டுமல்ல — GitHub repos, Kubernetes clusters, Datadog monitors, Cloudflare DNS — எல்லாவற்றையும் ஒரே Terraform codebase-ல் manage செய்யலாம்!
Chapter 08 · Intermediate
Workspaces
Multiple environments from one codebase
What are Workspaces?
Workspaces allow multiple isolated state files from the same configuration. Each workspace (dev/staging/prod) gets its own state. Same code, different infrastructure sizes per environment.
Workspace = same code, different environments. dev workspace-ல் dev state file, prod workspace-ல் prod state file. ஒரே .tf files-ஐ பயன்படுத்தி multiple environments manage!
# Workspace commands $ terraform workspace list # show all workspaces $ terraform workspace new dev # create dev $ terraform workspace new staging $ terraform workspace new prod $ terraform workspace select prod # switch to prod $ terraform workspace show # current workspace name
# Use workspace to control resource sizes locals { env = terraform.workspace instance_sizes = { dev = "t3.micro" # cheap — developer testing staging = "t3.medium" # medium — QA testing prod = "t3.xlarge" # powerful — live users } } resource "aws_instance" "app" { instance_type = local.instance_sizes[local.env] tags = { Environment = local.env Name = "app-${local.env}" } }
🚕 Real-world: Swiggy / Ola Multi-env
Swiggy has 4 environments: dev (devs test features), QA (testing team), staging (production mirror), prod (live users with 50M+ orders). All use identical Terraform code, different workspaces. Prod = 100 EC2 instances; dev = 2. Same codebase — workspace handles the difference!

Swiggy — dev, QA, staging, prod — four environments. Same Terraform code, different workspaces. Prod-ல் 100 servers, dev-ல் 2 servers. Workspace மட்டும் மாற்றினால் போதும்!
Chapter 09 · Advanced
Backend & Remote State
Team collaboration, state locking, Terraform Cloud
S3
AWS S3 + DynamoDB (Most Common)
S3 stores the state file (encrypted, versioned). DynamoDB table provides state locking — only one person can apply at a time, preventing conflicts. Free tier friendly for most teams.
S3-ல் state file, DynamoDB-ல் locking. ஒரே நேரத்தில் 2 பேர் apply செய்தால் conflict வராது — DynamoDB lock prevent செய்யும்.
TFC
Terraform Cloud (HCP Terraform)
HashiCorp's fully managed solution. Handles state, locking, remote plan/apply, secret variables, policy enforcement, audit logs, and a web UI. Free tier for up to 5 users.
HashiCorp-இன் managed service. State, locking, CI/CD, secrets, policy — எல்லாமே ஒரே platform-ல். 5 users வரை free.

Chapter 11 · Advanced
Best Practices
Write production-grade Terraform that teams can maintain
①
Structure Your Files
Separate concerns: main.tf (resources), variables.tf (var definitions), outputs.tf (outputs), versions.tf (provider pins), terraform.tfvars (values).
Code separate files-ல் வை: main.tf, variables.tf, outputs.tf, versions.tf. ஒரே file-ல் எல்லாம் போட்டால் maintain செய்வது கஷ்டம்.
②
Never Hardcode Secrets
Use environment variables, AWS Secrets Manager, HashiCorp Vault, or SSM Parameter Store for passwords and API keys. Add *.tfvars and .terraform/ to .gitignore.
Passwords, API keys — .tf files-ல் hardcode செய்யாதீர்கள். Environment variables, AWS Secrets Manager use செய்யுங்கள்.
③
Remote State Always (in Teams)
Local state in teams = inevitable disaster. Use S3+DynamoDB or Terraform Cloud. Enable S3 bucket versioning for state recovery if something goes wrong.
Team-ல் local state use செய்யாதீர்கள். S3+DynamoDB அல்லது Terraform Cloud mandatory. S3 versioning enable செய்யுங்கள்.
④
Tag Every Resource
Add tags to every resource: Environment, Team, Project, ManagedBy=Terraform, CostCenter. Enables billing allocation and faster troubleshooting.
Every resource-க்கும் tags add செய்யுங்கள். Environment, Team, CostCenter tags மூலம் billing track செய்யலாம்.
⑤
Pin Provider & Terraform Versions
Always pin: version = "~> 5.0" not "latest". Breaking changes in providers can silently destroy infrastructure if you auto-upgrade.
Provider version pin செய்யுங்கள். Latest use செய்தால் breaking changes வந்து infrastructure break ஆகலாம்.
⑥
Always fmt + validate Before Commit
Run terraform fmt (auto-format) and terraform validate before every commit. Add as pre-commit hooks and CI pipeline steps.
Commit-க்கு முன் terraform fmt மற்றும் terraform validate. Pre-commit hooks-ல் add செய்யுங்கள் — automatic quality check.

# Dynamic blocks — avoid repetitive ingress rules resource "aws_security_group" "web" { dynamic "ingress" { for_each = var.ingress_rules content { from_port = ingress.value.port to_port = ingress.value.port protocol = "tcp" cidr_blocks = ingress.value.cidrs } } } # for_each — create multiple unique resources resource "aws_s3_bucket" "environments" { for_each = toset(["dev", "staging", "prod"]) bucket = "shopDesi-${each.key}-assets" } # Conditional — prod gets multi-AZ, dev doesn't resource "aws_db_instance" "main" { multi_az = terraform.workspace == "prod" ? true : false deletion_protection = terraform.workspace == "prod" ? true : false backup_retention_period = terraform.workspace == "prod" ? 7 : 1 } # Terraform native test (v1.6+) — .tftest.hcl run "vpc_has_correct_cidr" { command = plan assert { condition = aws_vpc.main.cidr_block == "10.0.0.0/16" error_message = "VPC CIDR block must be 10.0.0.0/16" } }
TG
Terragrunt — DRY Wrapper
Wrapper around Terraform that eliminates repeated backend configs and provider definitions across multiple environments. Manages dependencies between Terraform modules automatically.
Terragrunt — Terraform wrapper. Multiple environments-க்கு same config duplicate செய்யாமல், DRY principle follow செய்ய help செய்யும்.
🔒
Policy as Code — OPA / Sentinel
Enforce org policies before resources are created. "No public S3 buckets", "Prod must use encrypted EBS", "All resources need cost-center tag" — enforced automatically in CI/CD.
Policy as Code — "public S3 create செய்யக் கூடாது", "prod-ல் encryption mandatory" — CI/CD-ல் automatically enforce. Manual compliance check தேவையில்லை.
💰
Infracost — Cost Estimation in PRs
Shows cloud cost breakdown before terraform apply. Integrates with CI/CD to post "this PR will increase AWS bill by ₹15,000/month" directly in PR comments. Prevents surprise bills.
Infracost — apply-க்கு முன்னே "இந்த infra ₹15,000/month cost ஆகும்" என்று PR comment-ல் காட்டும். Budget shock தவிர்க்கலாம்!
📊
Atlantis — GitOps for Terraform
Self-hosted Terraform automation bot. Runs terraform plan on every PR and terraform apply on merge. Full audit log of who applied what and when. Popular in large engineering orgs.
Atlantis — GitHub PR-ல் comment செய்தால் automatically plan/apply run செய்யும். யார், என்ன, எப்போது apply செய்தார் என்று full audit log.
