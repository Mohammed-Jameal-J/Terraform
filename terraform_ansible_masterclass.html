Chapter 00 · Beginner
What is Terraform?
Definition, purpose, the big picture
Definition — English
Terraform is an open-source Infrastructure as Code (IaC) tool by HashiCorp. You write code (HCL) describing the cloud infrastructure you want — servers, databases, networks — and Terraform builds it. No manual clicking in the AWS console ever again.
Terraform என்பது HashiCorp-இன் open-source கருவி. Cloud-ல் servers, databases, networks — எல்லாவற்றையும் code எழுதி create செய்யலாம். AWS console-ல் manually click செய்யத் தேவையில்லை.
Real-world analogy
Blueprint for a house: Instead of telling the contractor step-by-step each time, you write a blueprint once and reuse it. Terraform is that blueprint for cloud infrastructure — write once, deploy 100 times across AWS, Azure, GCP.
Key terms
IaC — Infrastructure as Code HCL — HashiCorp Config Language Declarative — describe end state Idempotent — same result every run
Declarative vs imperative: Bash = tell every step (imperative). Terraform = say "I want this to exist" and it figures out the steps (declarative).
Declarative vs Imperative: Bash-ல் step-by-step சொல்வது imperative. Terraform-ல் "இந்த infrastructure வேண்டும்" என்று சொல்வது declarative.

Chapter 02 · Beginner
Terraform Architecture
Core, State, Providers — how it works internally
1
Terraform Core
The brain. Reads .tf config files, compares desired state vs current state, builds a dependency graph, orchestrates the apply process in correct order.
Core = Terraform-இன் brain. Config files படிக்கும், plan செய்யும், dependency graph build செய்யும்.
2
State file (terraform.tfstate)
Terraform's memory. A JSON file mapping your code to real-world resources. Tracks every resource ID, attribute, and metadata. Next apply compares this to detect changes.
State file = Terraform-இன் நினைவாற்றல். "AWS-ல் i-0abc123 instance create செய்தோம்" — இங்கே track ஆகும்.
3
Provider plugins
Adapters that talk to each cloud API. AWS provider → AWS API. 3000+ providers: GitHub, Datadog, Cloudflare, Kubernetes, MongoDB — manage everything as code.
Provider = cloud API-ஓடு பேசும் translator. AWS, Azure, GCP மட்டுமல்ல — GitHub, Datadog, K8s — எல்லாவற்றிற்கும் providers உள்ளன.
4
Configuration files (.tf)
Your infrastructure blueprint in HCL. Contains resources, variables, outputs, data sources. Terraform reads these to know what to create.
HCL language-ல் எழுதப்படும் blueprint files. Resources, variables, outputs — எல்லாமே இங்கே.
Chapter 03 · Beginner
Core Lifecycle
init → plan → apply → destroy
①
terraform init
Downloads providers, sets up backend, prepares working directory. Always the first command in any project. Safe to run multiple times.
Providers download, backend setup. புதிய project-ல் முதல் command.
②
terraform plan
Dry run. Shows what will be created (+), modified (~), or destroyed (-). Never applies anything. Always run before apply to review changes.
Preview மட்டும். Actually எதுவும் create/destroy ஆகாது. Apply-க்கு முன் எப்போதும் plan run செய்யுங்கள்.
③
terraform apply
Executes the plan. Creates/modifies/destroys infrastructure. Asks confirmation. Updates state file. Use -auto-approve in CI/CD.
Actual-ஆக infrastructure create/change/delete. "yes" type செய்தால் proceed. State file update ஆகும்.
④
terraform destroy
Removes all Terraform-managed resources. Useful for dev/test to save costs. Be very careful in production!
எல்லாவற்றையும் delete செய்யும். Dev/test-ல் cost save-க்கு useful. Production-ல் மிகவும் கவனமாக!
Chapter 04 · Intermediate
HCL Language
Syntax, variables, resources, outputs, data sources
# Provider terraform { required_providers { aws = { source = "hashicorp/aws", version = "~> 5.0" } } } provider "aws" { region = "ap-south-1" } # Variable variable "instance_type" { type = string default = "t3.micro" } # Resource resource "aws_instance" "web" { ami = data.aws_ami.latest.id instance_type = var.instance_type tags = { Name = "web-${var.environment}", ManagedBy = "Terraform" } } # Output output "server_ip" { value = aws_instance.web.public_ip }
HCL explained
HCL is human-readable, JSON-compatible, supports comments, functions, loops, and conditions. Think of it as JSON with superpowers — designed for humans to write and machines to parse.
HCL = Human-readable config language. JSON போல், ஆனால் comments, loops, conditions support. மனிதர்கள் எளிதாக படிக்கலாம்.
Chapter 05 · Intermediate
State Management
terraform.tfstate — Terraform's memory
Local state — bad for teams
Only on your laptop
Team can't share it
No locking — 2 people apply = disaster
Lost if disk fails
Remote state — team-ready
Stored in S3 / GCS / Blob
Whole team shares one state
DynamoDB locking — no conflicts
Versioned and backed up
# S3 remote backend with DynamoDB locking terraform { backend "s3" { bucket = "my-terraform-state" key = "prod/terraform.tfstate" region = "ap-south-1" encrypt = true dynamodb_table = "terraform-state-lock" } }
Never manually edit terraform.tfstate! Corrupt state = Terraform loses track of all infrastructure. Always use terraform state commands.
terraform.tfstate-ஐ manually edit செய்யாதீர்கள் — state corrupt ஆனால் infrastructure track இழக்கப்படும்.
Chapter 06 · Intermediate
Modules
Reusable infrastructure components — DRY principle
What is a module?
A module is a container for multiple resources — like a function in programming. Write once, call many times with different parameters. Enables DRY infrastructure code.
Module = programming-ல் function போல். ஒரு module-ல் பல resources define செய்யலாம். பலமுறை different parameters-உடன் call செய்யலாம்.
# Calling a module from the Terraform Registry module "vpc" { source = "terraform-aws-modules/vpc/aws" version = "~> 5.0" name = "my-vpc-${var.environment}" cidr = "10.0.0.0/16" azs = ["ap-south-1a", "ap-south-1b"] private_subnets = ["10.0.1.0/24", "10.0.2.0/24"] public_subnets = ["10.0.101.0/24", "10.0.102.0/24"] enable_nat_gateway = true } # Reference module output in another resource resource "aws_instance" "app" { subnet_id = module.vpc.private_subnets[0] }
Chapter 07 · Advanced
Workspaces & Best Practices
Multiple environments, remote backends, production-grade code
# Workspaces — same code, different environments $ terraform workspace new dev $ terraform workspace new prod $ terraform workspace select prod # Use workspace in resources locals { sizes = { dev = "t3.micro", prod = "t3.xlarge" } } resource "aws_instance" "app" { instance_type = local.sizes[terraform.workspace] }
Structure files properly
main.tf (resources) · variables.tf (var definitions) · outputs.tf · versions.tf (provider pins) · terraform.tfvars (values)
Never hardcode secrets
Use AWS Secrets Manager, HashiCorp Vault, or environment variables. Add *.tfvars to .gitignore.
Tag every resource
Environment, Team, Project, ManagedBy=Terraform, CostCenter. Required for billing and troubleshooting.
Run fmt + validate before commit
terraform fmt (auto-format) and terraform validate (syntax check). Add as pre-commit hooks.
Chapter 08 · Beginner · Ansible
What is Ansible?
Agentless configuration management — automate what's inside your servers
Definition — English
Ansible is an open-source configuration management and automation tool by Red Hat. It connects to servers over SSH and runs tasks — install software, configure files, start services — without installing any agent on the target machine. You write Playbooks (YAML files) describing what the server should look like.
Ansible என்பது Red Hat-இன் open-source configuration management கருவி. Servers-க்கு SSH மூலம் connect ஆகி — software install, files configure, services start — எல்லாமே automatically செய்யும். Target machine-ல் எந்த agent-உம் install செய்ய தேவையில்லை. Playbook என்ற YAML files-ல் "server எப்படி இருக்க வேண்டும்" என்று எழுதுவோம்.
Real-world analogy — நிஜ உலக உதாரணம்
Chef in a restaurant: Terraform builds the kitchen (walls, stoves, sinks — infrastructure). Ansible is the recipe the chef follows inside that kitchen — install the right software, configure the right settings, ensure the right services are running.

சமையலகார உதாரணம்: Terraform = கிச்சன் கட்டும் (infrastructure). Ansible = அந்த கிச்சனில் என்ன செய்ய வேண்டும் என்ற recipe (software install, config). இரண்டும் வேறு வேறு கட்டங்களில் செய்யும் — together ஒரு complete setup.

Chapter 09 · Beginner · Ansible
Ansible Architecture
Control node, managed nodes, inventory, playbooks, modules
1
Control node
The machine where Ansible is installed and run from. Your laptop, a bastion host, or a CI/CD server. Ansible only needs to be installed here — not on the target servers.
Control node = Ansible install செய்யப்பட்ட machine. உங்கள் laptop அல்லது CI/CD server. Target servers-ல் எந்த software-உம் install தேவையில்லை.
2
Managed nodes (inventory)
The target servers Ansible configures. Listed in an inventory file (hosts). Can be grouped: [web], [db], [app]. Ansible connects to them over SSH automatically.
Managed nodes = Ansible configure செய்யும் target servers. Inventory file-ல் IP addresses list செய்யப்படும். Groups-ஆக organize செய்யலாம்: [web], [db], [app].
3
Playbook
A YAML file describing what to do on which servers. Contains one or more plays. Each play targets a group of hosts and runs a list of tasks in order.
Playbook = YAML file. "எந்த servers-ல், என்ன செய்ய வேண்டும்" என்று describe செய்யும். Plays மற்றும் tasks-ஆல் composed.
4
Modules
Built-in tools Ansible uses to perform tasks. apt (install packages), copy (copy files), service (start/stop services), template (render Jinja2 templates), shell (run commands). 3000+ modules available.
Modules = Ansible-இன் built-in tools. apt (packages install), copy (files copy), service (services manage), template (config files render). 3000+ modules ready-made.
5
Roles
A structured way to organise playbooks into reusable units — like Terraform modules. A "nginx" role contains tasks, templates, handlers, and variables all related to Nginx setup.
Role = Terraform module போல். Reusable playbook structure. "nginx role" = nginx-க்கு தேவையான எல்லா tasks, templates, variables ஒரு folder-ல் organized.
Chapter 10 · Intermediate · Ansible
Playbooks & YAML Syntax
Writing your first playbook — tasks, handlers, variables, templates
Playbook structure
A playbook is a YAML file with one or more plays. Each play specifies hosts and a list of tasks. Tasks use modules. Handlers run only when notified (e.g., restart Nginx only if config changed).
Playbook = YAML file. ஒரு playbook-ல் பல plays இருக்கலாம். ஒவ்வொரு play-லும் hosts மற்றும் tasks list. Handlers = config change ஆனால் மட்டும் service restart செய்யும்.
# inventory.ini — list your servers [web] 192.168.1.10 192.168.1.11 [db] 192.168.1.20 [all:vars] ansible_user=ubuntu ansible_ssh_private_key_file=~/.ssh/id_rsa
# site.yml — basic playbook --- - name: Configure web servers hosts: web become: true # sudo vars: nginx_port: 80 app_name: "shopDesi" tasks: - name: Install Nginx ansible.builtin.apt: name: nginx state: present update_cache: true - name: Copy Nginx config ansible.builtin.template: src: nginx.conf.j2 dest: /etc/nginx/nginx.conf notify: Restart Nginx # handler triggered - name: Ensure Nginx is running ansible.builtin.service: name: nginx state: started enabled: true handlers: - name: Restart Nginx ansible.builtin.service: name: nginx state: restarted
# Run the playbook $ ansible-playbook -i inventory.ini site.yml # Dry run — check what will change $ ansible-playbook -i inventory.ini site.yml --check # Run on specific group only $ ansible-playbook -i inventory.ini site.yml --limit web # Run specific tag $ ansible-playbook -i inventory.ini site.yml --tags nginx
Real-world: Swiggy server config
Swiggy has 200 web servers. After Terraform creates them, Ansible runs one playbook: install Node.js, copy the app config, set environment variables, start the app service, enable log rotation. All 200 servers configured identically in under 10 minutes.

Swiggy (தமிழ்): 200 web servers-க்கு Ansible ஒரே playbook run செய்யும். Node.js install, config copy, service start — 10 minutes-ல் எல்லா servers ready. Manual work zero!
Chapter 11 · Intermediate · Ansible
Roles & Ansible Galaxy
Reusable role structure — like Terraform modules for config
Role directory structure
Roles organise related tasks, variables, templates, and handlers into a standard folder structure. Install community roles from Ansible Galaxy (like the Terraform Registry for Ansible).
Role = Terraform module போல் reusable. Standard folder structure-ல் tasks, variables, templates organize செய்யப்படும். Ansible Galaxy-ல் community-built roles free-ஆக available.
# Role folder structure roles/ nginx/ tasks/ main.yml # main task list handlers/ main.yml # restart handlers templates/ nginx.conf.j2 # Jinja2 config template vars/ main.yml # role variables defaults/ main.yml # default values (overridable) meta/ main.yml # role metadata + dependencies
# Using roles in a playbook --- - name: Setup all servers hosts: all become: true roles: - common # base config for all servers - nginx # web server - nodejs # app runtime - monitoring # Prometheus node exporter # Install from Ansible Galaxy $ ansible-galaxy install geerlingguy.nginx $ ansible-galaxy install geerlingguy.nodejs # Or use requirements.yml --- roles: - name: geerlingguy.nginx version: "3.2.0"
Ansible Galaxy has 10,000+ community roles — pre-built playbooks for Nginx, MySQL, Docker, Java, Python, Redis, and more. Use them instead of writing from scratch, exactly like Terraform Registry modules.
Ansible Galaxy-ல் 10,000+ community roles. Terraform Registry போல் — already written roles use செய்யுங்கள். Time save ஆகும்!
Chapter 12 · Advanced · Ansible
Terraform + Ansible Together
The complete automation combo — provision then configure
Why use both?
Terraform is great at creating cloud resources but not at configuring what's running inside them. Ansible is great at configuring servers but not at provisioning cloud infrastructure. Together they cover the full stack.
Terraform = cloud resources create செய்வதில் expert. Software config-ல் weak. Ansible = servers configure செய்வதில் expert. Cloud provisioning-ல் weak. Together = complete automation!
# Step 1: Terraform creates EC2, outputs IP resource "aws_instance" "web" { ami = data.aws_ami.latest.id instance_type = "t3.medium" key_name = "my-key" tags = { Name = "web-server" } } output "web_ip" { value = aws_instance.web.public_ip } # Step 2: Pass Terraform output to Ansible inventory # (via local-exec provisioner or CI/CD script) resource "null_resource" "ansible_run" { depends_on = [aws_instance.web] provisioner "local-exec" { command = "ansible-playbook -i '${aws_instance.web.public_ip},' site.yml" } }
# Better: dynamic inventory (reads from AWS directly) $ ansible-playbook -i aws_ec2.yml site.yml # aws_ec2.yml — dynamic inventory plugin plugin: amazon.aws.aws_ec2 regions: - ap-south-1 filters: tag:ManagedBy: Terraform instance-state-name: running keyed_groups: - key: tags.Environment prefix: env

Chapter 13 · Master · Ansible
Master Level Ansible
Vault · dynamic inventory · molecule testing · AWX/Tower
Ansible Vault — secrets management
Encrypt sensitive data (passwords, API keys) in your playbooks and variable files. Vault-encrypted files can be safely committed to Git. Decrypt at runtime with a password or key file.
Ansible Vault = secrets encrypt செய்யும். Passwords, API keys-ஐ encrypt செய்து Git-ல் commit செய்யலாம். Runtime-ல் decrypt ஆகும்.
Dynamic inventory
Instead of a static hosts file, query AWS/Azure/GCP for running instances in real time. When Terraform adds new servers, Ansible automatically discovers them — no manual inventory updates.
Static inventory file-க்கு பதிலாக AWS-ல் இருந்து live-ஆக servers query செய்யும். Terraform new servers create செய்தால் Ansible automatically discover செய்யும்.
Molecule — test your roles
Testing framework for Ansible roles. Spins up a Docker container or VM, runs your role, then verifies the expected outcome. Write tests before roles — TDD for infrastructure configuration.
Molecule = Ansible roles-க்கு testing framework. Docker container-ல் role run செய்து, expected outcome verify செய்யும். Infrastructure config-க்கும் TDD possible!
AWX / Ansible Tower — web UI for teams
Web-based UI for running Ansible playbooks across an organisation. Role-based access control, scheduling, audit logs, credential management. AWX is the open-source version; Tower is enterprise (Red Hat).
AWX = Ansible-இன் web UI. Team members browser-ல் playbooks run செய்யலாம். RBAC, scheduling, audit logs — enterprise-grade Ansible management.
